# Systematic Diagnose Issue

## 概述

随机猜测浪费时间，还会引入新 bug。快速补丁掩盖了潜在问题。

**核心原则：** 先定位问题根因，不尝试修复。诊断完成后输出诊断报告，修复工作由其他 agent 或人工处理。

**铁律：**
```
DIAGNOSE FIRST, FIX LATER — 但本 SKILL 只做 DIAGNOSE
```

## 何时使用

适用于任何需要找到根因的场景：
- 测试失败
- 生产 bug
- 异常行为
- 性能问题
- 构建失败
- 集成问题
- E2E 测试不稳定
- 前后端接口对接问题

**特别要使用当：**
- 已经尝试过多种修复但没有效果
- 上一次的修复没有生效
- 不完全理解问题时
- 需要明确问题归属（前端/后端/测试本身）

## 三阶段流程（只做诊断）

必须完成每个阶段后才能进入下一阶段。

### Phase 1: 证据收集

**在尝试任何结论之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误码
   - 区分：真正的问题 vs 连带问题

2. **稳定复现**
   - 能可靠触发吗？
   - 确切步骤是什么？
   - 发生频率？
   - 如果无法稳定复现 → 收集更多环境信息

3. **检查最近变更**
   - 什么变更可能导致这个问题？
   - Git diff、最近的 commit
   - 新依赖、配置变更
   - 多人协作时的代码冲突

4. **多组件系统的边界证据收集**

   **当系统有多个组件时（如：前端 → API → 后端 → 数据库）：**

   **在提出结论之前，逐层收集证据：**

   ```
   对于每个组件边界：
     - 记录请求发出时的数据
     - 记录边界处的实际数据
     - 验证环境/配置传递是否正确
     - 标记第一次数据异常的位置
   ```

   **运行测试或手动操作一次，收集证据显示哪里出了问题**
   **不要猜测，证据会说话**

### Phase 2: 模式分析

**基于证据形成假设：**

1. **对比 working 案例**
   - 在相同代码库中找到类似功能
   - 对比：什么能 work，什么坏了？

2. **识别关键差异**
   - working 和不 working 的区别是什么？
   - 列出每个差异，不管多小
   - 不要假设"那个不重要"

3. **定位根因层级**

   | 症状层级 | 说明 | 例子 |
   |---------|------|------|
   | 数据层 | 传递的数据本身有问题 | ID 不存在、格式错误、null |
   | 逻辑层 | 数据处理逻辑错误 | 过滤条件错误、计算错误 |
   | 接口层 | API 契约不匹配 | 字段名不同、类型不同 |
   | 配置层 | 环境或配置问题 | 端口、路径、权限 |
   | 依赖层 | 外部服务问题 | 第三方 API 不可用 |

4. **理解依赖和假设**
   - 这个模块依赖什么？
   - 它做什么假设（类型、存在性、格式）？
   - 假设是否被满足？

### Phase 3: 验证假设

**用最小化方式验证根因：**

1. **形成单一假设**
   - 清楚声明："根据证据，我认为 X 是根因，因为 Y 发生时 Z 出错"
   - 写下来

2. **设计验证实验**
   - 需要什么最小信息来验证这个假设？
   - 能否通过日志、断点、打印来确认？

3. **执行验证**
   - 收集能证明或反驳假设的证据
   - 不要修改代码（只收集信息）

4. **如果假设被推翻**
   - 形成新假设
   - 回到步骤 1 继续收集证据
   - 记录被推翻的假设，避免重复

## 诊断输出格式

完成三阶段后，输出标准诊断报告：

```markdown
## 诊断报告

### 问题描述
[一句话描述核心问题]

### 发生条件
- 触发步骤 1
- 触发步骤 2
- 触发步骤 3

### 根因分析

**根因位置**: [前端/后端/数据库/配置/测试本身]
**根因描述**: [清晰描述根本原因]

**证据**:
1. [证据 1]
2. [证据 2]
3. [证据 3]

### 排除的假设
- [被推翻的假设 1] — 证据：[为什么排除]
- [被推翻的假设 2] — 证据：[为什么排除]

### 建议
[后续处理建议，可选]
```

## 红旗 - 停止并重新审视

如果你发现自己想：
- "大概是 X，我直接修复"
- "先试试改 X 看有没有效"
- 在没有充分证据的情况下提出修复方案
- 在完成证据收集前跳到结论
- 跳过复现步骤

**这些意味着：停。回到 Phase 1。**

## 常见错误

| 错误 | 问题 | 正确做法 |
|------|------|---------|
| 看到错误直接修复 | 可能是连带问题，不是根因 | 完整追踪数据流 |
| 假设 X 导致 Y | 没有验证就下结论 | 形成假设后验证 |
| 一次检查多个地方 | 无法隔离问题 | 一次一个变量 |
| 复现不成功就放弃 | 可能需要更多环境信息 | 收集更多证据 |
| 把测试问题归咎于代码 | 可能是测试本身有问题 | 检查测试代码 |
